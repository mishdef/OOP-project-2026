# Курсовий проєкт: управління АЗС (Gas Station Management System)

Курсовий проєкт з дисципніни **"Об'єктно-орієнтоване програмування"**.  
Розроблено студентом Національного аерокосмічного університету "ХАІ" групи 621п, **Стадніковим М.Г.**

---

## Про Проєкт

Цей застосунок є сучасною Desktop-системою для автоматизації робочих процесів на автозаправній станції (АЗС). Основна мета — забезпечення швидкого та інтуїтивного UX для касира, а також надання потужних інструментів аналітики та адміністрування для керівника.

Проєкт реалізовано на платформі .NET із використанням архітектурного патерну MVVM та суворим дотриманням принципів SOLID та ООП.

---

## Ключовий Функціонал

### Рольова модель доступу

- **Касир (Cashier UI):**
    - Управління наливом пального в реальному часі (моніторинг статусів колонок: _Вільна, Зайнята, Обслуговується_).
    - Швидкий продаж супутніх товарів (кава, фастфуд, автотовари) за допомогою інтегрованого пошуку та категорій.
    - Програма лояльності: сканування, валідація та автоматичний розрахунок накопичувальних бонусових балів.
    - Формування чеків та розрахунок решти.
- **Адміністратор (Admin Dashboard):**
    - Управління ціновою політикою (пальне та товари) в реальному часі.
    - Контроль залишків пального в резервуарах (Танках) із системою критичних сповіщень.
    - Управління обліковими записами персоналу (CRUD).
    - Генерація та експорт фінансових звітів (Z-звіти).

## Архітектура та ООП Принципи

### Реалізація базових концепцій ООП:

1. **Інкапсуляція:** усі поля класів є приватними. Доступ до них реалізовано через властивості (`properties`) із вбудованою бізнес-валідацією (наприклад, перевірка максимальної місткості резервуара перед оновленням об'єму).
2. **Спадкування:** створено базовий абстрактний клас `Product`, від якого успадковуються конкретні класи `Fuel` (Пальне) та `Good` (Супутній товар).
3. **Поліморфізм:** використовується динамічне зв'язування. Розрахунок вартості кошика (`Subtotal`) відбувається поліморфно для різних типів продуктів, що знаходяться у загальному списку `List<Product>`.
4. **Абстракція:** бізнес-логіка повністю відірвана від конкретних реалізацій завдяки інтерфейсам (`IPumpService`, `IBonusService`, `IAuthService`).

### Принципи SOLID та Патерни:

- **SRP (Single Responsibility):** кожен сервіс відповідає лише за одну задачу. Логіка авторизації (`AuthService`) ніяк не перетинається з логікат резервуарів (`TanksService`).
- **OCP (Open/Closed):** завдяки патерну **Strategy**, розширити асортимент новими категоріями послуг можна без зміни коду існуючих платіжних модулів.
- **LSP (Liskov Substitution):** класи `Fuel` та `Good` можуть повністю замінити батьківський `Product` без порушення коректності роботи програми.
- **ISP (Interface Segregation):** інтерфейси розділені за функціоналом (наприклад, статистика винесена окремо від генерації звітів).
- **DIP (Dependency Inversion):** впроваджено Dependency Injection (DI). Усі сервіси ін'єктуються у ViewModel через конструктори.
- **Паттерн Builder:** використовується в `OrderBuilder` для покрокового формування чеків складної структури.
